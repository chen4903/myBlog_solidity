12.stack 1024 level

发表于2022-12-31|更新于2023-01-06|03.solidity进阶

stack 1024 level

前置知识

  • code：部署合约时，data字段是存放合约内容的空间，即专门存放智能合约二进制源代码的空间。
  • storage：Storage是一个可读、可写、可修改的持久化存储空间，也是每个合约持久化存储数据的地方。Storage是一张巨大的map，一共有2^256个slot，每个slot有32个字节。合约中的“状态变量”将根据其特定类型存储在这些插槽中。
  • stack：Stack就是所谓的“运行栈”，用来保存EVM指令的输入输出数据。可以免费使用，没有gas消耗，用于存储函数的局部变量数量限制为16个。栈的最大深度为1024，每个单元为32字节。
  • args：也叫calldata，是一个只读的可寻址空间，用于存储函数调用参数。与stack不同的是，如果要使用calldata中的数据，必须手动指定offset和读取的字节数。
  • memory：内存是一个简单的字节数组，主要是存储运行时的数据，以及给内部函数传递参数。基于32byte的寻址和扩展。

storage是永久存储在区块链上的地方。如果你想操作storage中的数据，你可以将它复制到内存中。然后，所有内存代码都在堆栈上执行。Stack 的最大深度为 1024 个元素，支持 256 位的字长。

当您定义局部变量时，它存储在内存中，然后压入堆栈以执行。

简介

EVM不是寄存器机而是堆栈机，所以所有的计算都在称为堆栈的数据区域上进行。

There are 2 stacks in Ethereum, both Normal stack and Call stack are created for every new transaction.

  • nornal stack：专门给EVM来存储临时数据的stack
    • Normal stack is used to store data while executing the contract like function arguments, local variables etc. 【用来存放变量、引用、方法或是方法返回值，用来存储函数内存中的数据】
    • You should still avoid excessive recursion, as every internal function call uses up at least one stack slot and there are only 1024 slots available.1024 slots, each slot 32 bytes
    • 它不做任何运算，只保存临时数据
  • call stack：调用合约内部的函数不会增加层数，每一个外部调用（call, callcode, delegatecall, staticcall）都有自己的call stack，用来存储参数、返回值和局部变量等。
    • 每一个局部变量都会占据一个slot，比如局部变量是bytes1，并不会将32个bytes1拼凑在一起占用一个slot
    • 1024 slots, each slot 32 bytes
    • 合约无法使用与查看call stack，合约只知道mg.sender和tx.origin，合约无法知道中间是否还有其他合约调用。但是我们在remix的debug中可以看到call调用其他合约的堆栈和内存，这是remix自己搞的功能，它可以访问所有的内容，然而EVM本身是没有这个功能的。想要看到中间调用了哪些外部合约，要进行contract trace。
    • 调用链越深，你需要的插槽就越多，如果你走得太深，你最终会用超1024个插槽，然后报错。通常 1024 个就足够了，除非出现无限递归或者循环过多
    • call stack 也叫execution stack，它会执行一系列操作码和运算，其数据来自normal stack。
    • 假如合约A调用自身的方法b()，call stack 不会增加。用户B调用b()，会增加。合约A调用其他合约的函数，会增加。

设计原因

我认为将stack设计为1024个slot的原因：

  • 具有固定大小使得 EVM 的整体模型更加简单且易于实现
  • 如果它非常大，那么执行合约会更昂贵（即需要更多内存）。 1024 是一个非常保守的值，以尽可能安全
  • EVM 的设计方式往往会使更大的堆栈变得无用。EVM 只能访问堆栈中前16个slot。因此，即使您有一个 4096 slot的的堆栈，也只能够访问前16个，除非你不断pop，才可以访问到下面更深层次的内容

stack访问限制

对stack的访问仅限于顶端：

  • 可以将最顶端的前 16 个元素之一复制到堆栈的顶部：
    • 操作码【PUSH n】，将第n个slot的内容放入堆栈中，n从 1 到 16
    • 操作码【DUP n】，复制第n 个堆栈项，n从 1 到16
  • 可以将最顶端的元素与前 16 个元素之一交换
    • 操作码【SWAP n】，交换第 1 个和第n 个堆栈项，其中 n从 1 到 16
  • 其他的操作就是正常的stack操作：取栈顶的元素，计算，压回栈顶
    • 正常的pop和push操作

例子

攻击者合约通常会生成一个循环的嵌套调用序列，以创建一个几乎满了的的call stack，然后调用受害者的合约函数，该函数将在进一步调用时失败。如果受害者的合约没有正确处理异常，攻击者可能会成功攻击。

在上面给出的例子中，Governmental合约允许用户在合约中进行投资，最后一个进行投资且 1 分钟内没有发生另一笔投资的人将获得头奖并将剩余的以太币发送给合约所有者。

这里的Governmental是易受攻击的合约，被名为Hack的恶意合约所利用。本质上，Governmental是一个蜜罐陷阱，收集以太币并只支付给所有者。Hack通过递归调用自身并依次增长堆栈来利用漏洞堆栈大小限制。导致堆栈被放大。

由于实际获胜者未获得报酬，每次执行此攻击时，这种失败都会使Governmental合约的余额增加。为了让所有者从Governmental 中吸取汇集的以太币，所有者只需允许另一轮正确终止。
