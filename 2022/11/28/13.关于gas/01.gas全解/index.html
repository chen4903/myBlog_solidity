gas全解

简介

Gas 是用于测量在以太坊区块链上执行特定操作所需的计算工作量的单位，即以太坊网络上的计算单位。我们可以简单的把 Gas 理解为汽油，其保证了以太坊网络的正常运行。在以太坊区块链上，执行写入操作都需要支付一定的 Gas 费用。

以太gas限额是指用户进行一笔交易可以消耗的最大gas量。

涉及智能合约的交易更加复杂，需要更多的计算能力来执行。因此，与发送付款等简单交易相比，这些交易需要更高的气体限额。

特点

  • 以太坊汽油费的标准限额是 21,000
  • 第一次执行一个非view、pure方法的时候，消耗的gas会多一点：可能是因为要写入链中，生成插槽或者分配内存需要额外的gas，但之后之后就会覆盖插槽或内存，消耗的gas就少了且固定
  • 计算一个方法所用的gas，不可以仅仅算方法体内每一步的gas总和
    • 执行一个view方法，进行debug：
      • 还没开始执行，就会执行vm trace stop（操作码执行的次数），会消耗掉一部分的gas
      • 执行方法体
      • 等等
  • Sending message to any non-existing account costs more than sending message to an existing account. That also applies to the accounts of the precompiled contracts.To not pay higher costs of the message you need to create the destination account before. That is the reason why some test/private genesis blocks creates the accounts for the precompiled contracts by setting theirs balances to 1 wei, e.g. testnet genesis block.

  • 将气体限制设置得太高是可以的——EVM 将退还未使用的部分。但是将 gas 限制设置得太低可能会导致用户损失一些 ETH 并导致他们的交易被拒绝。

    • 例如，如果用户为 ETH 转账设置 50,000 的以太气体限额，则 EVM 将消耗 21,000 并退还剩余的 29,000。但是，如果有人将气体限制设置为 20,000，而交易需要 21,000 个单位，则 EVM 在尝试完成交易时可能会消耗 20,000 个气体单位，但交易不会完成。在这种情况下，用户将保留他们尝试发送的 ETH，但他们的 20,000 个气体单位将丢失，因为 EVM 消耗了它以试图完成失败的交易。

  • 自己在代码Call里面设置上限Gas ，然后再调用MetaMask，最后还是以MetaMask上限Gas为准。至于最终真实使用多少Gas，以矿工为准。如果MetaMask上限Gas低于旷工的期望值，可能就不会被打包上链。如果MetaMask上限Gas远大于实际使用额Gas，多出来的GAS也不会退回来，当作小费，优先把你的交易打包上链。

名词解释

  • base：由上一个区块决定的基础gas，

  • max：用户自己决定的最大gas价格，一般应该大于基础费+消费。如果小于，就会使用max来作为Gas Price

  • max Priority：消费，最少1 Gwei

  • gas price：Gas Price 是以太坊内消耗 1 个 Gas 对应多少的 Gwei，当然交易发送方可以自定义愿意支付的每单位 Gas 价格。比如交易发送方的一笔交易需要耗费 10 Gas，交易发送者愿意支付 3 Wei/Gas，交易的成本总价就是 30 Wei。在我们使用的钱包（如 MetaMask）里都会有调整 Gas 费的高级选项，Gas Price 越高，打包的优先级就越高。
  • gas limit：Gas Limit 是消耗 Gas 的限制单位，即交易发送方在完成每笔交易时最多能使用的用于执行交易的 Gas 量。如果没有 Gas Limit 做限制，可能会导致交易发送方的账户余额因错误操作而消耗完，Gas Limit 是安全机制，防止把账户中所有 ETH 消耗掉。
  • 固有成本gas：
    • 发送一笔交易的成本包括两部分：固有成本、执行成本。而固有成本由交易的负载决定：
      • 创建智能合约的负载是创建智能合约的EVM 代码
      • 调用智能合约函数的负载是执行消息时输入的数据
      • 两个账户之间转账的负载为空

我们假设N_zeros 代表的是交易负载中字节为 0 的字节总数，N_nonzeros代表交易负载中字节不为 0 的字节总数，根据黄皮书 6.2 章节关于执行的描述，可以得到固有成本。

计算规则

  • Transaction Free = Gas Price * Usage by Txn
    • 72.923038038 * 21000 = 1531383.798798
  • Gas Price = Base + Max Priority
    • 72.923038038 = 71.923038038 + 1
  • Burnt = Base * Usage by Txn
    • 71.9230380380 * 2100 = 1510383.798798
  • Txn savings = (Max - Gas Price ) * Usage by Txn
    • (74.345484113 - 72.923038038) * 2100 = 29871.367575
  • 同一个区块的base都是一样的

gas优化

  • Using a custom error instance will usually be much cheaper than a string description, because you can use the name of the error to describe it, which is encoded in only four bytes. A longer description can be supplied via NatSpec which does not incur any costs.
  • Compared to regular state variables, the gas costs of constant and immutable variables are much lower. For a constant variable, the expression assigned to it is copied to all the places where it is accessed and also re-evaluated each time. This allows for local optimizations. Immutable variables are evaluated once at construction time and their value is copied to all the places in the code where they are accessed. For these values, 32 bytes are reserved, even if they would fit in fewer bytes. Due to this, constant values can sometimes be cheaper than immutable values.
  • The hash of the signature of the event is one of the topics, except if you declared the event with the anonymous specifier. This means that it is not possible to filter for specific anonymous events by name, you can only filter by the contract address. The advantage of anonymous events is that they are cheaper to deploy and call. It also allows you to declare four indexed arguments rather than three.
  • 获取链上数据的gas(100 gas)比获取内存中的gas(3 gas)多得多。因此，如果方法中需要多次用到同一个链上状态数据，可以先将它获取用一个变量A接收，然后之后的每一处用到的地方用变量A代替即可。例子：UniswapV2Pair.sol
  • 使用immutable和const标注不需要改变的变量
  • 在内存缓存中读取变量：从存储成本中的变量读取gas。当我们第一次访问一个变量时，它会花费2100 gas，随后的访问也会花费100 gas。如果我们有一个数组并且想要对数组元素执行一些计算。将数组的长度存储在一个变量中更好
  • 使用calldata
  • +1操作：示例列出了四个合约，它们显示了我们可以将数字变量增加 1 的方式。合约Four在运行时更有效地从批次中取出。要在增加变量时节省更多费用，请用第四种方式
  • 使用view：调用一个view方法不需要消耗gas。但是如果在一个方法A中，调用一个view的方法B，那么方法B将消耗gas
  • 使用uncheck处理整数溢出：在 Solidity0.8.0版本之前，整数溢出和下溢检查是通过使用该SafeMath库执行的。从 Solidity0.8.0版本之后，编译器会为我们进行检查。这造成额外费用gas。如果我们知道我们将在合约内部执行的数学运算不会上溢或下溢，我们可以告诉编译器不要检查操作中的上溢或下溢。合约OverFlow有两个函数用于在合约内部设置两个变量。编译器setNumberOne检查并防止数字上溢或下溢。在第二个函数setNumberTwo中，我们告诉编译器要注意它的事情，不要检查溢出或下溢。使用unchecked意味着不会检查代码块是否有上溢或下溢。如果我们知道我们的数学是安全的，我们可以通过使用unchecked
  • 使用revert代替require：我们require用来检查一个语句是否是true. 如果语句为假，则交易将被还原，剩余的 gas 将返还给用户。由于 Solidity 中有自定义错误语法，我们可以使用该revert语句来发出自定义错误。使用revert而不是require更省气。您可以运行下面的代码来查看使用时gas节省的数量，revert而不是require.
  • 避免在内存变量中填写过大的数据：当一个memory变量中的参数大于32kb的时候，消耗的gas将会平方级增加。我们应避免传入过大的数据给memory变量
  • 在进行任何状态更改之前，始终将您的require语句放在函数的顶部，以便如果require语句失败，剩余gas的将在还原时退还给用户。使用&&或||与require语句进行比较操作时；您应该首先放置操作中较便宜的部分，这样如果它失败，编译器将不会比较第二部分，从而节省gas
