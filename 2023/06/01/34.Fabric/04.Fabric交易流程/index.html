Fabric交易流程

假定各节点已经提前颁发好证书，且已正常启动，并加入已经创建好的通道  在已经实例化了的链码通道上从发起一个调用交易到最终记账的全过程

  • 客户A从客户B 处购买萝卜
  • Peer A为客户A的区块链节点，Peer B 为客户B的区块链节点
  • 链码：实现购买萝卜的价格和交易逻辑
  • 链码背书策略：Peer A和Peer B必须同时背书

1.创建交易提案并发送给背书节点

  • 应用程序构造好交易提案请求后，选择背书节点执行并进行背书签名
  • 交易提案由消息头和消息结构组成，并且添加了调用者的签名信息
  • 背书节点会根据签名信息验证其是否是一个有效的消息
  • 背书节点是链码背书策略里指定的节点
  • 有些背书节点可能离线，其他背书节点可能拒绝对交易进行背书，或不背书 ，应用程序可以尝试使用其他可用的背书节点来满足策略
  • 应用程序以何种顺序给背书节点发送背书请求是没有关系的，正常情 况下背书节点执行后的结果是一致的，只有背书节点对结果的签名不 一样

2.背书节点模拟交易并生成背书签名

  • 在收到交易提案后会进行一些验证
    1. 交易提案的格式是否正确
    2. 交易是否提交过（重复攻击保护）
    3. 交易签名是否有效（通过 MSP )
    4. 交易提案的提交者在当前通道上是否已授权有写权限
  • 验证通过后，会根据当前账本数据模拟执行链码中的业务逻辑并生成读写集
  • 在模拟执行时，账本数据不会更新
  • 对这些读写集进行签名成为提案响应，然后返回给应用程序
  • 返回的提案响应中包含了读写集、背书节点签名以及通道名称等信息

读写数据集

3.收集交易的背书

  • 应用程序收到提议响应后会对背书节点签名进行验证
  • 如果链码只进行账本查询，应用程序会检查查询响应，但不会将 交易提交给排序服务节点
  • 如果链码对账本进行 Invoke 操作， 则须提交交易给排序服务 进行账本更新
  • 应用程序会在提交交易前判断背书策略是否满足
  • 如果应用程序没有收集到足够的背书就提交交易了，记账节点在 提交验证阶段会发现交易不能满足背书策略，标记为无效交易

4.构造交易请求并发送给排序服务节点

  • 应用程序接收到所有的背书节点签名后，根据背书签名调用 SDK 生成交易，广播给排序服务节点
  • 生成交易过程需要确认所有的背书节点的执行结果完全一致，再 将交易提案 、 提案响应和背书签名打包生成交易
  • 应用程序可以把生成的交易信封内容发送给任意选择的几个排序 服务节点

5.排序节点对交易进行排序并生成区块

  • 接收网络中所有通道发出的交易信息，获取通道名称，按各个 通道上交易的接收时间顺序对交易信息进行排序，生成区块
  • 不读取交易的内容，如果在生成交易信封内容的时候伪造了交 易模拟执行的结果，排序节点不会发现，但会在最终的交易验 证阶段校验出来并标记为无效交易

6.排序节点广播区块给组织的主节点

排序节点生成区块以后会广播给通道上不同组织的主节点

7.记账节点验证区块内容并写入区块

  • 记账节点接收到的是排序服务节点生成的区块
  • 验证区块交易的有效性，提交到本地账本
  • 再产生一个生成区块的事件，监听区块事件的应用程序可以进行后续的 处理
  • 如果接收到的区块是配置区块，则会更新缓存的配置信息

记账节点验证区块内容并写入区块

World state: (k1,1,v1), (k2,1,v2), (k3,1,v3), (k4,1,v4), (k5,1,v5)

  • T1 -> Write(k1, v1’), Write(k2, v2’)

    • T1 通过验证，没有读操作， k1 和 k2 的世界状态更改 为 (k1,2,v1’), (k2,2,v2’)

  • T2 -> Read(k1，1), Write(k3, v3’)

    • T2 没有通过验证，因为读了被T1修改过的 k
  • T3 -> Write(k2, v2’’)
    •  T3 通过验证，没有读操作， k2 的世界状态更改为 (k2,3,v2’’)
  • T4 -> Write(k2, v2’’’), read(k2，1)
    • T4 没有通过验证，因为读了被T1 修改过的 k2
  • T5 -> Write(k6, v6’), read(k5，1)
    • T5 通过验证，读过 k5, 没有被任何前序交易修改过

8.在组织内部同步最新的区块
