06.Fabric共识机制

发表于2023-01-28|更新于2023-01-28|34.Fabric

Fabric共识机制

简介

  • Fabric共识机制
    • Fabric共识算法：Kafka算法/RAFT算法
    • Fabric共识规则：基于投票的三阶段规则

Fabric区块链框架业务通过执行两个框架达成单独的共识过程：交易排序、交易验证 共识过程分解为三个阶段：背书阶段、排序阶段和验证阶段

  • 背书阶段：客户端节点根据背书策略，选择背书节点，发送交易提案，背书节点调用智能 合约执行模拟交易，执行完成后，经过签名背书，返回给客户端节点
  • 排序阶段：排序节点接收已经签名背书的交易，确定交易顺序，将排好序的交易打包成区 块，分发给其他组织主节点，排序节点不关心交易是否正确，只负责排序和打 包区块
  • 验证阶段：区块存储和交易验证不冲突，区块存储前进行交易验证，区块状态是世界状态

Fabric共识流程

背书阶段

背书节点对客户端发来的交易预案进行合法性检验

  1. 客户端应用程序将交易请求打包成为交易提案后，根据背书策略发送 给指定的背书节点
  2. 各背书节点接收到交易提议后，首先进行一些检查和签名的验证，然 后独立地模拟执行指定的链码，并不会将数据记录到账本中
  3. 执行完成后生成一个提议结果，并调用交易背书系统链码（ESCC） 对执行结果进行签名
  4. 将提议结果返回给客户端应用程序
  5. 当客户端应用程序收集到足够多的提议结果后，背书阶段完成

排序阶段

排序节点的作用

排序节点的核心过程

  • 排序节点收到交易后，验证数据 的客户端签名是否正确
  • 验证通过后，将交易发给kafka集 群对应的topic
  • 排序节点中的每个通道都在kafka 上监听对应的消息，kafka将消息 存放到对应topic上之后，会将消 息广播给通道上的所有排序节点
  • 因为各个排序节点的消息都是由 kafka按照相同顺序发送的，因此， 这个过程也实现了消息的排序

排序节点的核心过程

  • 客户端通过 gRPC 连接发送交易信息到排序节点的 Broadcast() 接口
  • 排序节点收到请求后，提取消息进行解析、检查，通过检查后封装为 Kafka 消息， 通过 Produce 接口发送到 Kakfa 集群对应的 topic 分区中
  • 当消息数达到设定数量或消息尺寸过大，或超时时间，则发送分块消息到 Kafka
  • Kafka 集群维护多个 topic 分区（对应多个通道），一旦写入分区，任何排序节 点看到的都是相同的消息队列

接受提交的被认可的交易并进行排序，确保交易顺序的一致性

  1. 排序节点接收到来自于多个客户端应用程序的交易提议结果
  2. 对所有提交的交易进行排序，按照排序节点接收到交易的时间
  3. 将交易分批打包进区块中，形成一个统一的共识结果，这种机制保 证了Fabric不会出现账本的分叉
  4. 当等待足够时间或区块满足大小后，将打包好的区块发送给特定通 道中的所有节点

验证阶段

获取有序事务块并验证其结果的正确性，包括检查背书策略和重复提交攻击

  • 通道中的节点在接收到排序节点广播的区块后，每个节点都按照相同的 方式独立处理接收到的区块，保证账本的一致性

    1. 通过验证服务链码(VSCC)检查交易是否满足背书策略
    2. 检查账本当前状态是否与提议结果生成时一致
    3. 通过检查的成功交易将被更新到账本中
    4. 构造事件消息，向注册了事件监听的客户端应用程序发送事件消息

  • 为了校验交易的正确性，共识的建立依赖于智能合约层

  • 智能合约层定义了商业逻辑来如何验证交易的有效性  智能合约层根据特定的策略与约定来确认每一笔交易都是有效的

  • 无效的交易会被拒绝，并在块中剔除

  • 潜在的校验失败主要分为以下两种：

    • 语法错误：无效输入、未验证的签名、重复的交易等，这类交 易应该被丢弃
    • 逻辑错误：这类错误应该需要定义策略决定是继续处理还是终止执行，如导致重复交易的交易应该终止执行

可插拔共识机制

  • Fabric 应用程序可以根据不同的交易背书、排序和验证模型要求， 实现支持对3个阶段的可拔插共识服务

  • 排序节点通过gRPC服务提供两个 API 接口

    • broadcast（blob）：客户端调用此函数在通道中广播任意的消息 blob，客户端向排序服务发送交易请求
    • deliver（seqno，prevhash，blob）：排序服务调用此函数给节点发 送blob消息，包含非负整数的序列号 seqno 和最近一次消息的哈希 prevhash

  • Fabric 正式版本中支持两种共识算法类型：

    • Solo：单节点共识，整个Fabric网络只有一个排序节点，主要用于 测试模式
    • Kafka：分布式消息队列，整个Fabric网络的共识由 Kafka 集群实 现，Kafka实现了对于Fabric网络中所有的交易请求进行排序服务

  • Hyperledger 项目团队正在开发其它排序共识插件

    • 简化拜占庭容错算法（Simplified Byzantine Fault Tolerance， SBFT）
    • 蜜罐拜占庭容错算法（Honey Badger of BFT）
