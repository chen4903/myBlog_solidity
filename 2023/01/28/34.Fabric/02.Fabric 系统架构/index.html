Fabric 系统架构

简介

Fabric是超级账本代码库的第一个提案

Fabric的系统架构

应用层

  • 标准的 gRPC API接口
  • 高级语言的SDK：Golang 、 Node.js 、Java 、 Python 等。开发人员可以利用 SDK 开发基于区块链的应用
  • 事件：各个节点之间达成共识需要较长的执行时间，采用异步通信模式。 事件模块可以在触发区块事件或者链码事件的时候执行预先定义的回 调函数

应用程序角度

  • 身份管理
    • 用户注册和登录系统后，获取到用户注册证书
    • 其他所有的操作都需要与用户证书关联的私钥进行签名
    • 消息接收方首先会进行签名验证，才进行后续的消息处理
    • 网络节点同样会用到颁发的证书，比如系统启动和网络节点管理等都会 对用户身份进行认证和授权

  • 账本管理

    • 授权的用户可以查询账本数据
    • 查询方式：区块号查询区块 、 区块哈希查询区块、交易号查询区块、 交易号查询交易，通道名称获取查询到的区块链信息

  • 交易管理

    • 账本数据只能通过交易执行才能更新，应用程序通过交易管理提交交易提案
    • 获取到交易背书以后，再给排序服务节点提交交易，然后打包生成区块
    • 利用用户证书本地生成交易号，背书节点和记账节点会校验是否存在重复交易
  • 智能合约
    • 实现“可编程的账本” ，通过链码执行提交的交易，实现基于区块链的智能合约业务逻辑
    • 只有智能合约才能更新账本数据，其他模块是不能直接修改状态数据

底层角度

  • 成员管理

    • 注册登录
      • 每个 MSP 都会建立一套根信任证书体系
      • 利用 PKI对成员身份进行认证，验证成员用户提交请求的签名
      • 结合 Fabric-CA 或者第三方CA 系统，提供成员注册功能
      • 对成员身份证书进行管理，例如证书新增和撤销
    • 属性证书
      • 注册证书（ ECert ）：用于用户身份
      • 交易证书（ TCert）：用于交易签名
      • TLS 证书（ TLS Cert）：用于TLS 传输

  • 共识服务

    • 在分布式节点环境下，要实现同一个链上不同节点区块的一致性，同时 要确保区块里的交易有效和有序
    • 共识机制由 3 个阶段完成
    • 客户端向背书验证节点提交提案进行签名背书  客户端将背书后的交易提交给排序服务节点进行交易排序，生成区 块和排序服务
    • 广播给记账节点验证交易后写入本地分布式账本
    • 网络节点的 P2P 协议采用的是基于 Gossip 的数据分发，以同一组 织为传播范围来同步数据，提升网络传输的效率

  • 链码服务

    • 智能合约的实现依赖于安全的执行环境，确保安全的执行过程和用户数 据的隔离
    • Fabric 采用 Docker 管理普通的链码，提供安全的沙箱环境和镜像文件 仓库
    • Docker优势
      • 容易支持多种语言的链码，扩展性很好
    • Docker劣势
      • 对环境要求较高，占用资源较多，性能不高等
      • 存在与 Kubernetes 、 Rancher 等平台的兼容性问题

  • 安全和密码服务

    • Fabric 专门定义了一个 BCCSP ( BlockChain Cryptographic Service Provider），使其实现密钥生成、哈希运算、签名验签、加密解密等功能
    • BCCSP 是一个抽象的接口，为上层模块提供密码学服务
      • SW：国际标准加密算法的软实现，默认值
      • PKCS11：国际标准加密算法的硬实现
      • 目前社区在实现国密的算法
