04.比特币交易签名及验证

发表于2023-01-28|更新于2023-01-28|32.比特币

比特币交易签名及验证

比特币验证

  • 账号所有权验证：如何在不泄露密码的前提下，证明自己对某个账户的所有权呢？
  • 比特币所有权验证：如何在不泄露密码的前提下，证明自己对要交易的比特币的所有权呢？
  • 交易所有权验证：如何在不泄露密码的前提下，证明某个交易的确是你创建的呢？

在去中心化的前提下，每个节点独立进行验证

去中心化验证

  • 比特币交易脚本语言是没有中心化主体的
    • 没有任何中心主体能凌驾于脚本之上
    • 没有中心主体会在脚本被执行后对其进行保存
  • 执行脚本所需信息都必须包含在脚本中
  • 一个脚本能在任何系统上以相同的方式执行
  • 如果你的节点验证了一个脚本，可以确信比特币网络中的每一 个其他节点也将验证这个脚本，这意味着一个有效的交易对每 个人而言都是有效的
  • 这种结果可预见性是比特币系统的一项至关重要的良性特征

去中心化验证

  • A要从账户X中转账一笔钱给账户Y，就需要向全网广播一条消息

  • 消息中不能包含账户X的密码，否则就意味着账户的控制权泄露给了全网 的用户

  • 网络中的其他节点收到消息后，需要能够验证A，确实拥有账户X的所有权

账号所有权验证

  • 由A向全网发布消息，广播的消息中需要包括：

    • 原始交易信息： 打款账号地址，收款账户地址，打款金额

    • 交易信息摘要： 原始交易信息经过Hash算法后得到的一条摘要信息

    • 签名： 签名= sign (私钥, 交易信息摘要

  • 其他节点验证A是否拥有账户X：拥有正确的私钥，即拥有账户X的所有权

    • 获取打款账号的公钥：在原始交易信息中，获取打款者的账户地址
    • 使用校验函数进行验证：交易信息摘要 = verify(签名, 公钥

交易的构成

  • Input：计划花费的UTXO的来源
    • txid : 引用的UTXO所在的那笔交易的ID
    • vout : 引用的UTXO所在交易的输出中的序号
    • scriptSig : 解锁脚本，包含付款人对本次交 易的签名()和付款人公钥()
  • Output：计划生成几个UTXO，分别给谁， 每个UTXO里面有多少BTC
    • value : 比特币数量
    • n : UTXO序号（从0开始）
    • scriptPubkey : 锁定脚本，包含命令和收款 人的公钥哈希（)

锁定脚本

  • 锁定脚本是一个放置在输出上面的花费条件，它指定了今后花费这笔输出 必须要满足的条件

  • 锁定脚本往往含有一个公钥或比特币地址（公钥哈希值）

交易输出

解锁脚本

  • 解锁脚本是一个满足被锁定脚本在一个输出上设定的花费条件的脚本

  • 解锁脚本是每一笔比特币交易输入的一部分

  • 解锁脚本包括

    • 由支付者私钥生成的签名

    • 支付者的公钥（）

  • 签名用来证明支付者同意把这笔钱给接收者

  • 签名用来证明支付者拥有这笔钱

  • 签名用来证明这个交易是支付者创建的且没有被修改过

交易输入

交易签名

签名简介

  • 签名的输入
    1. 待签名的交易数据：输入、输出
    2. 引用的UTXO相关信息：交易ID、序号、锁定脚本
    3. 支付者的私钥：
    4. 签名类型
  • 签名的输出
    • scriptSig（解锁脚本）：签名（）、支付者的公钥（<PubK(B)>）

交易验证的目的：

  1. 金额是否充足？在本地的UTXO集合中查找到该笔交易的所有输入所在的区块，判 断支付金额是否充足
  2. 是否存在双花问题？在上一步查找的区块高度之后，是否存在对这些UTXO的支付行为， 从而保证这些UTXO没有被双花
  3. 是否是自己的钱？验证加锁脚本，包括验签、持有者地址比较等
  4. 收款方是否合法？验证锁定脚本，判断收款方是否是合法地址

验证简介

签名验证的目的

  • 证明交易所引用的UTXO的确属于付款人，具体到本次交易，就是证明交易001的序号为0的UTXO的确是发给B的
  • 证明交易的所有数据的确是付款人提供的，且未被修改过。具体到本次交易，就是证明B的确创建了交易002，且交易内的数据未 被修改过

签名验证的方法

  • 每个验证节点会通过执行解锁脚本解锁对应UTXO的锁定脚本来验证 一笔交易
  • 交易的每个输入都包含一个解锁脚本，并引用了之前存在的 UTXO
    1. 验证软件将复制解锁脚本
    2. 检索输入所引用的 UTXO
    3. 从该 UTXO 复制锁定脚本
    4. 依次执行解锁和锁定脚本
    5. 如果解锁脚本满足锁定脚本条件，则输入有效
  • 所有输入都是独立验证的，作为交易总体验证的一部分

验证过程

验证意义

  • 步骤1~6的意义：
    • 证明交易001的序号为0的UTXO的确是发给B的  用B提供的公钥（）进行双哈希（HASH160），然后与锁 定脚本中的公钥哈希（）作对比，相同则返回成功
    • 公钥哈希（）是A在创建交易时根据B的地址生成的， 就是B的公钥经过双哈希运算得来的，所以只要提供了B的公钥，验证肯 定是成功的
    • 步骤1~6 就相当于A把1BTC发给了B的邮箱，B拿把钥匙打开了邮箱， 证明了B确实拥有这1BTC，但并没有证明拿钥匙的人就是B
  • 步骤7的意义：证明了B确实拥有那把钥匙，B的确创建了交易002， 且交易内的数据未被修改过
